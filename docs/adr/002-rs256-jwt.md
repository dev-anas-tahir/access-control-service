# ADR-002 — RS256 JWT with JWKS Distribution

**Status:** Accepted  
**Date:** 2025-05

## Context

The platform needs a token scheme that:
1. Allows other services (catalog, order) to verify tokens without calling IAM on every request.
2. Supports key rotation without redeploying consumers.
3. Is revocable before expiry for logout and compromise scenarios.

## Decision

Use RS256-signed JWTs with:
- RSA private key held exclusively by `iam-service` for signing.
- RSA public key distributed via `GET /.well-known/jwks.json` (JWKS endpoint).
- Short access token TTL (15 min default) to limit revocation window.
- JTI-based revocation stored in Redis for the remaining lifetime of the token.

Access tokens carry `sub`, `username`, `roles[]`, `permissions[]`, `is_super_user` so downstream services can make authorisation decisions without a network call to IAM.

## Consequences

- HS256 (shared secret) rejected: distributing a secret to all consumers is operationally risky.
- Consumers can verify tokens offline after a one-time JWKS fetch (with caching).
- Key rotation requires a JWKS update and a brief overlap window while old tokens expire — documented in [`../runbooks/key-rotation.md`](../runbooks/key-rotation.md).
- Revocation is best-effort within the 15-minute window; longer-lived tokens would require per-request revocation checks against IAM.
